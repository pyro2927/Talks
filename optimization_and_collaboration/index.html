Code Style

Decided by majority, enforced via EditorConfig.

Majority can be outweighed by a senior developer. Easy to combined mulitple types of styles into a single 
.editorconfig
. Allows developers to use whatever tools they want while still adhearing to the same code standards.

.editorconfig

# Unix-style newlines with a newline ending every file
[*]
end_of_line = lf
insert_final_newline = true

# Matches multiple files with brace expansion notation
# Set default charset
[*.{js,py}]
charset = utf-8

# 2 space indentation
[*.rb]
indent_style = space
indent_size = 2
Line endings can be an actual issue, as we've seen with Windows developers trying to run cross-platform code. Dreaded ^M

Share the Load

Make it easy for everyone. Don't be the sole gatekeeper.

GitHub has everyone deploy on their first day. Strive for low-friction contributions. Setup Capistrano on day 1 and let junior devs and non-devs deploy. Also works as a least common-denominator. If you make stuff so easy a non-dev can use it, you'll only help yourself in keeping things easy.

Travis-CI

Travis runs on each commit and alerts us in Slack if something breaks.

.travis.yml

language: ruby
rvm:
  - 2.2.1
bundler_args: "--without production"
script:
  - rake db:test:prepare
  - bundle exec rspec spec
before_script:
  - psql -c "CREATE USER evolve WITH SUPERUSER" -U postgres
before_install:
  - sudo apt-get update
install:
  - sudo apt-get install default-jre-headless
  - bundle install --without production
Using Travis-CI makes sure that we don't have any one-off dependancies met because tests are run in a fresh container each time. Scripts must be complete!
