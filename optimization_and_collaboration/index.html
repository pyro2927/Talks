Share the Load

Make it easy for everyone. Don't be the sole gatekeeper.

Gatekeeper == bottlekneck GitHub has everyone deploy on their first day. Strive for low-friction contributions. Setup Capistrano on day 1 and let junior devs and non-devs deploy. Also works as a least common-denominator. If you make stuff so easy a non-dev can use it, you'll only help yourself in keeping things easy. For a while I was the only dev who knew how to restart our production servers. "Hit by a bus" stuff.

PR Handling

Assign them!

(and feel free to ocassionally nag remind)

One question I get often is "what is the best way to handle PRs?". Hold people accountable, give them direct responsibility. Even if there are a few people on a team, assign our PRs. Threaten to merge after a day or two. "I know it works, I'm going to merge it in tomorrow if no one else reviews it"
Bad PR What API? What is considered public data?
Good PR vs bad PR Good to include a picture, inline is even nicer.
Great PR! Include documentation on how to run tests, even if it's just 
bundle exec rspec spec
We complain all the time about something being thrown "over the wall", don't do that to another dev.

Code Style

Decided by majority, enforced via EditorConfig.

Majority can be outweighed by a senior developer. Easy to combined mulitple types of styles into a single 
.editorconfig
. Allows developers to use whatever tools they want while still adhearing to the same code standards.

.editorconfig

# Unix-style newlines with a newline ending every file
[*]
end_of_line = lf
insert_final_newline = true

# Matches multiple files with brace expansion notation
# Set default charset
[*.{js,py}]
charset = utf-8

# 2 space indentation
[*.rb]
indent_style = space
indent_size = 2

# tabs for HTML & erb
[*.{html,erb}]
indent_style = tab
Line endings can be an actual issue, as we've seen with Windows developers trying to run cross-platform code. Dreaded ^M

EditorConfig Plugins

Emacs, Vim, Visual Studio, SublimeText and TextMate Supported by all main editors. If you're fancy enough to use something more obscure, you're probably good enough to just follow the code style-guide.

Code for the Edge Case

Find situations where code WON'T work.

Turn it into a game. Sometimes it is easier to find a problem in someone else's code rather than your own.

Seeds are Your Friend

Populate the development database with a metric shit-ton of data.

Coded a "past" view, once we seeded 5000 records, it took over 2 minutes to render the page. Forces devs (FE and ruby) to consider controller views. Things like pagination become clear. You'll find areas you need to add caching or other helpful things.

Don't Trample Each Other's Work

Tests will ensure nothing breaks.

Setup a system that alerts you immedietly upon breakage. Notifications and automated tests are wonderful. Rails developers don't need to be told to use tests, but once you start interacting with developers you might find one who hasn't heard of tests before.
There are developers who haven't heard of or used tests before, don't look down on them.

Super Tests!

Rspec, FactoryGirl, VCR, Simplecov, and Ladle

Basic setup, simplecov to ensure code coverage, Ladle allows quick and easy LDAP usage without cluttering actual LDAP installation.

SimpleCov

We don't strive for 100%, but we try to get high and make sure the potential problematic areas well.

Travis-CI

"How many people have had a breaking change merged into master?" Keep notifications automated and asyncronous. Troublesome area is keeping the noise-to-signal ratio low. For now we keep a separate channel in Slack. Integrate it into your workflow. Sometimes not running tests can be a slippery slope

Get Notified

Travis runs on each commit and alerts us in Slack if something breaks. W/ Slack notifications, it is kinda a "set it and forget it" model. Enjoy using Travis because tests run in "clean" containers each time, and keeps things compartmentalized.

.travis.yml

language: ruby
rvm:
  - 2.2.1
bundler_args: "--without production"
script:
  - rake db:test:prepare
  - bundle exec rspec spec
before_script:
  - psql -c "CREATE USER evolve WITH SUPERUSER" -U postgres
before_install:
  - sudo apt-get update
install:
  - sudo apt-get install default-jre-headless
  - bundle install --without production
Using Travis-CI makes sure that we don't have any one-off dependancies met because tests are run in a fresh container each time. Scripts must be complete! before_script and install sections end up looking a lot like our Ansible script

Keep Tests Snappy!

Extra tid-bit, keep tests fast. Fast tests will get run more often.
